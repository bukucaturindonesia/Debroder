# v1.0 Implementation Report

Date: 2026-07-11

## Status

v1.0 local implementation is complete and buildable. Remote Supabase migration application is pending explicit owner approval because it changes the active connected project schema, grants, and RLS policies.

## Implemented

- Next.js 15 App Router foundation with strict TypeScript.
- Supabase product foundation migration for products, categories, variants, images, size master, and sellable SKU stock.
- Public product listing and `/produk/[slug]` detail route.
- Per-color gallery, color swatches, size selector, quantity validation, and disabled out-of-stock sizes.
- Multi-variant "Pilihan Anda" selection with duplicate combination merge.
- Versioned localStorage cart using `debroder_cart_v1`.
- Cart quantity edit, remove, clear, subtotal, and persistence after refresh.
- `/api/cart/revalidate` for latest stock and price checks before cart/WhatsApp flow.
- Basic WhatsApp payload using `NEXT_PUBLIC_SITE_URL` and optional `NEXT_PUBLIC_WHATSAPP_NUMBER`.
- PIM V2 admin surface with publish validation and guarded Supabase upsert endpoint.
- Focused tests for cart merge, stock validation, pricing, and publish validation.

## Known Limitations

- The active Supabase project has not received the migration yet.
- No production Supabase env values are stored in the repo.
- PIM V2 write endpoint is additive and safe by default, but should be exercised against a staging branch before production use.
- A local dev server was attempted but could not be left running because this shell lacks Node in PATH, detached Windows process launch failed, and the final foreground check was blocked by the approval/usage limit. The production build passes.
- v1.1 bulk/custom ordering remains intentionally out of scope until the v1.0 migration is applied and verified.

## Rollback Notes

The migration is additive and does not drop legacy data. If rollback is required before customer data exists, drop the v1.0 tables, helper function, policies, and enum types in reverse dependency order. If customer/order data exists, create a data-preserving rollback plan before removing any table.
