# DEBRODER v1.1 Implementation Report

## Summary

Implemented the v1.1 Bulk & Custom Ordering layer on top of the v1.0 product foundation without replacing the existing product, variant, size, stock, cart, or WhatsApp source of truth.

## Migration

Created:

- `supabase/migrations/20260711010000_v1_1_bulk_custom_ordering.sql`

## New Tables

- `product_price_tiers`
- `product_minimum_rules`
- `custom_services`
- `service_pricing_rules`
- `customer_uploads`
- `saved_configurations`
- `configuration_item_services`
- `quotation_drafts`
- `quotation_draft_items`

## Extended Runtime Data

- `products` query now reads `product_price_tiers` and `product_minimum_rules`.
- `CartItem` now supports price tier snapshot, services, upload references, item notes, service totals, and review state.
- Existing cart key remains unchanged: `debroder_cart_v1`.
- Autosave uses separate key: `debroder_product_configuration_v1`.

## Storage

Migration creates private bucket:

- `customer-designs`

Upload API:

- validates extension and MIME type
- limits file size to 20 MB
- sanitizes filename
- writes random private storage path
- stores metadata in `customer_uploads`
- returns a temporary signed preview URL
- supports delete by matching session token and storage path

## RLS

Migration enables RLS on all v1.1 public tables.

Public read:

- active price tiers
- active minimum rules
- active custom services
- active service pricing rules

Authenticated owner/staff access:

- own customer uploads
- own saved configurations
- own quotation drafts
- staff role access through existing `public.has_staff_role(...)`

Storage policies restrict `customer-designs` objects to owner or staff role. Service-role route handlers are server-only and do not expose the service key.

## Price Tier Implementation

- Tier is calculated from total quantity for the same `product_id`.
- Mixed colors and sizes are summed before pricing.
- Product page shows active tier, next tier, and remaining quantity.
- Server revalidation recalculates tier before add-to-cart and quotation draft.
- Admin PIM V2 can add/edit/remove tiers.
- Validator rejects overlap before save; migration also has a GiST exclusion constraint for active tier overlaps.

## Bulk Matrix

- Matrix rows come from active PIM V2 variants.
- Columns come from active sizes.
- Out-of-stock or unavailable combinations are disabled.
- Normal mode and bulk mode use the same `CartItem[]` draft state.
- Switching modes does not discard selections.

## Minimum Order

- `product_minimum_rules` supports product minimum, minimum for tier, and quotation threshold.
- UI shows specific minimum order errors.
- Add-to-cart blocks minimum order errors.
- Quotation route validates minimum order server-side.

## Services

- Service catalog is loaded from `custom_services` when Supabase is configured.
- Local fallback is used only when Supabase env is absent.
- Admin PIM V2 can add/edit service catalog rows.
- Supports pricing types: fixed per item, fixed per order, tiered, estimated, manual quote.
- Service notes and item notes are persisted in cart/configuration/quotation/WhatsApp.
- Partial service quantity is supported and validated.

## Quotation Draft

Created route:

- `POST /api/quotation-drafts`

The route stores full snapshots:

- product
- variant
- size
- SKU
- quantity
- price tier
- services
- file references
- notes
- contact information
- estimated/final totals

Admin page:

- `/admin/quotations`

## Persistence And Sharing

- Autosave writes versioned product configuration to `localStorage`.
- Refresh restores the saved configuration.
- Share link encodes configuration in the URL without private upload references.
- Private files are referenced by upload metadata/storage path, not permanent public URLs.

## Known Limitations

## Rollback Notes

To roll back v1.1 app code, remove the files listed above and restore the prior v1.0 versions. To roll back the database after applying the migration, drop v1.1 tables and the `customer-designs` bucket only after exporting any customer uploads/quotation drafts that must be retained.
